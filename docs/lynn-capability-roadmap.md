# Lynn 能力建设路线图

> 日期：2026-04-03
> 来源：Lynn 与 00 的能力缺口对齐讨论
> 定位：Lynn 下一阶段迭代的结构性参考

## 核心判断

Lynn 现在最缺的不是"更多功能"，而是三种底层能力：

1. **能自己动手** — 可授权的终端执行代理层
2. **能稳定找信息** — 适配中国网络环境的完整搜索链路
3. **能持续跟进任务** — 异步追踪、通知与任务恢复

这三件补上，后面的很多体验会一起抬起来。

---

## 能力缺口与优先级

### 第一梯队：终端执行 + 稳定搜索

#### 可授权终端执行

不是单纯少个 shell，而是少了"可授权、可回放、可持续放权"的执行代理层。

没有它，Lynn 永远像隔着玻璃指导你，而不是在电脑里真正干活。

需要做到：
- 可授权：用人话说清要干什么、有什么风险、授权范围多大
- 可回放：执行过程有日志，出问题可追溯
- 可持续放权：同类操作一次授权后自动放行，支持「本次会话 / 此项目长期」等粒度

#### 中国环境优先的稳定搜索

不是 web_fetch 能替代的。Lynn 缺的是"搜索 + 抓取 + 去重 + 结果置信度"的完整链路。

尤其要适配中国网络环境，不能默认 Google-first。

需要做到：
- 搜索源可配置（百度/Google/Bing 等）
- 抓取后自动去重、提取正文、标注来源
- 结果带置信度判断（官方文档 > 权威媒体 > 社区讨论 > 博客）
- 适配 JS 渲染页面的兜底方案

### 第二梯队：AST/规则化批量重构

决定 Lynn 是"会修文件"还是"会重构代码库"。

没有结构化重写，遇到跨 50+ 文件的改名、API 迁移、类型收敛，体验立刻掉回手工作坊。

需要做到：
- AST 级别的代码理解和变换
- semgrep 风格的模式匹配与批量替换
- 变更预览 + 确认 + 回滚
- 跨文件依赖分析（改名后自动更新引用）

### 第三梯队：异步追踪与通知

Lynn 从"对话工具"变成"住在电脑里的助手"的分水岭。

没有 watch/subscribe/notify，很多任务只能靠用户想起来再问一次，助手不算真的在持续工作。

需要做到：
- 支持对 URL / GitHub repo / 文件变更 / 价格等的监控订阅
- 条件触发时主动通知（推送到对话 / 系统通知）
- 通知带上下文摘要，不只是"有更新了"

### 第四梯队：项目级结构化记忆

现在的记忆是扁平文本，缺少结构化检索能力。但短期还没前三项致命，因为只要执行、搜索、异步还弱，记忆再强也只是更会记，却不一定更会干。

记忆的价值放大是乘法关系：执行能力是乘数，记忆是被乘数。乘数如果是零，被乘数多大结果都是零。

需要做到：
- 项目索引：每个项目有独立的记忆空间
- 时间线事件流：按时间排列的关键决策和变更
- 三维交叉召回：主题 / 决策 / 人物三个维度可交叉查询
- 相关性理由：召回时显式说明"这条记忆为什么和当前任务相关"

### 第五梯队：授权产品化 + 任务恢复

#### 授权产品化

现在的授权更像技术确认（yes/no），不像一个可信任助手的审批流。

应该做到：
- 说人话解释操作意图和风险
- 支持一次授权 / 本次会话 / 此项目长期授权
- 危险操作（删除、推送、付费）始终需确认
- 授权历史可审计

#### 任务恢复能力

长任务被打断后，Lynn 还不够会"接着上次做"。

需要做到：
- 任务快照：长任务的关键节点自动保存状态
- 工作树摘要：被打断后能快速回顾"做到了哪、还差什么"
- 待办状态恢复：重启对话后自动加载未完成任务
- 失败后续跑：失败后能从断点恢复而不是从头开始

## 建设顺序

| 优先级 | 能力 | 价值 |
|:---:|------|------|
| 1 | 可授权终端执行 | 从"隔着玻璃指导"到"真正干活" |
| 2 | 中国环境优先的稳定搜索 | 从"碰运气查资料"到"稳定获取信息" |
| 3 | AST/规则化批量重构 | 从"会修文件"到"会重构代码库" |
| 4 | 异步追踪与通知 | 从"对话工具"到"持续工作的助手" |
| 5 | 项目级结构化记忆 | 从"更会记"到"记的东西真正有用" |
| 6 | 授权产品化与任务恢复 | 从"技术可用"到"产品可信" |

## 实施路径

> 先用技能补齐能力缺口，再用底层算法逻辑提升长任务能力和回顾学习。
